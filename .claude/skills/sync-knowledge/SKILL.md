# sync-knowledge

The single owner of how `.ai/` is written and maintained. Both `write-prd` (which
declares a plan's `## Knowledge Base Impact`) and `execute-prd` (which calls this
at closeout) defer to the rules here — they are not repeated elsewhere.

`.ai/` exists so agents read the **map and the decisions** instead of re-deriving
the codebase from source every research pass. That only works if it stays small,
current, and decision-oriented.

## The rule: decision-oriented, not a wiki

> `.ai/` captures **decisions, constraints, invariants, and the map of where
> things live** — the non-obvious *why*. It does NOT restate what the code does.
> If a doc could be regenerated by reading the code, it doesn't belong here.

The test for any entry: *"Could a competent reader reconstruct this by reading
the code?"* If yes, leave it out. Echoing CLAUDE.md: no speculative content, no
dead entries, no comments that restate the code — only the non-obvious *why*.

**Prefer editing or retiring over appending.** The default failure mode of a
knowledge base is unbounded growth. When a change supersedes an entry, rewrite or
delete it — do not stack a new note on top of the stale one.

## What lives where

| File / dir        | Holds                                                                 |
| ----------------- | --------------------------------------------------------------------- |
| `index.md`        | The map: module → one-line responsibility → path → linked docs. Read first. |
| `architecture.md` | System shape: package boundaries, dependency direction, data flow.    |
| `decisions/<slug>.md` | One decision per file. The why + what was rejected.               |
| `patterns/<slug>.md`  | A reusable pattern — added only on the **2nd real use**, never speculatively. |

## index.md protocol

`index.md` is the highest-leverage file — it's what research subagents read to
skip straight to the relevant code.

- **Add a row** when a new module/package lands: its single-line responsibility,
  its path, and links to any `decisions/` or `patterns/` doc about it.
- **Edit a row** when a module's responsibility or path changes.
- **Retire a row** when the module is removed or merged — never leave a row
  pointing at a path that no longer exists.
- Keep rows terse. The index is a lookup table, not prose. If a cell wants a
  paragraph, that paragraph belongs in a `decisions/` or `architecture.md` entry
  the row links to.

## Decision-record shape

One decision per file in `decisions/`, named `<kebab-slug>.md`, kept short:

```markdown
# <Decision title>

**Decision:** <what was chosen, one or two sentences>

**Why:** <the non-obvious reasoning — constraints, trade-offs that forced it>

**Rejected:** <the alternatives considered and why they lost>

**Constraints it creates:** <invariants future code must respect, if any>
```

Record a decision only when the *why* isn't obvious from the code. A routine,
self-evident choice does not need a record.

## Sync routine (manual or called by execute-prd)

Given a completed change (and, when available, the plan's `## Knowledge Base
Impact` section as the starting checklist):

1. **Locate `.ai/`.** If it doesn't exist, there's nothing to sync — report that
   and stop.
2. **Reconcile against the change.** For each new concept, module, flow, or
   decision the change introduced:
   - update or add the `index.md` row,
   - add a `decisions/<slug>.md` if a non-obvious *why* was settled,
   - update `architecture.md` if a boundary, package, or flow changed,
   - add a `patterns/<slug>.md` only if this is the pattern's 2nd+ real use.
3. **Retire what's now stale.** Find entries the change superseded and rewrite or
   delete them. This step matters more than adding — it's what keeps `.ai/` from
   becoming a wiki.
4. **Apply the anti-wiki test** to everything you wrote: delete anything that
   merely restates code.
5. **Report** the `.ai/` files created / updated / retired, one line each.

When invoked from `execute-prd`, perform this in the merged branch after code has
landed, and hand the list of touched files back for the Final report.
